## 🎮 التحكم والأزرار

| الزر | الوصف | المفتاح |
|------|--------|---------|
| **⏭ Next** | الحدث التالي | مفيد للدراسة البطيئة |
| **⏮ Prev** | الحدث السابق | للعودة للخلف |
| **▶ Play** | تشغيل تلقائي | تتابع الأحداث تلقائياً |
| **⏹ Stop** | إيقاف التشغيل | للإيقاف المؤقت |
| **↻ Reset** | من البداية | لإعادة التشغيل |
| **السرعة** | 0.5 - 5 ثواني | تحكم كامل في السرعة |

## 🔍 أنواع الأحداث (Event Types)

### أحداث Buffer Cache الأساسية

#### 1️⃣ **HIT** (إصابة في الـ Cache) ✅
```
الحدث: BGET_HIT
التأثير: سريع جداً ⚡
الشرح: وجدنا البلوك في الذاكرة، بدون حاجة لقراءة من القرص

حالة البفر:
- Ref Count: +1 (عملية واحدة أكثر تستخدم البفر)
- Valid: لا يتغير (بيانات صحيحة)
- Locked: 0→1 (نقفل البفر للاستخدام)

🎯 ماذا يعني:
   النظام وجد البلوك بسرعة، الآن تستخدمه العملية
```

#### 2️⃣ **MISS** (إخفاق في الـ Cache) ⚠️
```
الحدث: BGET_MISS
التأثير: متوسط، يؤدي إلى تأخير ⏱️
الشرح: لم نجد البلوك في الـ Cache، محينا بفر قديم

حالة البفر:
- عدد المراجع: 0→1 (بفر فارغ جاهز)
- صحة البفر: ✓→✗ (بيانات قديمة أصبحت غير صالحة)
- القفل: 0→1 (قفل البفر)

🎯 ماذا يعني:
   يجب قراءة البيانات من القرص (سيكون هناك تأخير!)
```

#### 3️⃣ **FILL** (ملء من القرص) 📥
```
الحدث: BREAD_FILL
التأثير: بطيء جداً (هذا هو التأخير!) 🐢
الشرح: انتهينا من قراءة البلوك من القرص

حالة البفر:
- صحة البفر: ✗→✓ (البيانات الآن صحيحة!)
- Valid: 0→1 (يمكن الآن استخدام البيانات)

🎯 ماذا يعني:
   البيانات الآن جاهزة للاستخدام
```

#### 4️⃣ **WRITE** (كتابة إلى القرص) 💾
```
الحدث: BWRITE
التأثير: بطيء (I/O operation)
الشرح: العملية تريد حفظ التعديلات على القرص

حالة البفر:
- Locked: 1→1 (يبقى مقفولاً أثناء الكتابة)
- لا تتغير الحالات الأخرى

🎯 ماذا يعني:
   حفظ البيانات المعدلة بشكل دائم
```

#### 5️⃣ **RELEASE** (إفراغ البفر) 🔓
```
الحدث: BRELEASE
التأثير: سريع جداً
الشرح: انتهت العملية من استخدام البفر

حالة البفر:
- Ref Count: n→(n-1) (عملية أقل تستخدمه)
- Locked: 1→0 (فتح البفر)
- إذا Ref==0: البفر يصبح متاحاً لعمليات أخرى

🎯 ماذا يعني:
   البفر جاهز الآن للاستخدام من عملية أخرى
```

## 📊 الإحصائيات

### ماذا تعني النسب؟

```
📊 Hits = 50
   عدد المرات التي وجدنا البيانات في الـ Cache مباشرة ✅

📊 Misses = 10
   عدد المرات التي لم نجد البيانات وأضطررنا لقراءتها ❌

📊 نسبة الإصابة = Hits / (Hits + Misses) * 100
   = 50 / 60 * 100 = 83.3% ✨

🎯 الهدف:
   نسبة إصابة عالية = نظام فعال
   نسبة إصابة منخفضة = حاجة لتحسين الخوارزمية
```

## 🎓 سيناريوهات التعلم

### السيناريو 1: فهم HIT vs MISS

**الخطوات:**
1. اضغط "Play" بسرعة 5 ثواني (بطيء)
2. لاحظ الفرق:
   - **HIT**: تغيير بسيط في Ref Count فقط
   - **MISS**: تغييرات كبيرة في Valid و Locked

**النتيجة:**
✅ فهمت أن HIT أسرع من MISS

---

### السيناريو 2: تتبع تأثير FILL

**الخطوات:**
1. ابحث عن BGET_MISS في السجل
2. ثم اذهب للحدث التالي (BREAD_FILL)
3. شاهد كيف Valid يتغير من 0 إلى 1

**النتيجة:**
✅ فهمت عملية البحث والقراءة الكاملة

---

### السيناريو 3: LRU ونسيان البيانات القديمة

**الخطوات:**
1. ابحث عن BGET_MISS
2. لاحظ `old_blockno` - البلوك القديم الذي تم حذفه
3. هذا يوضح كيفية استبدال البفرات

**النتيجة:**
✅ فهمت نظام LRU (Least Recently Used)

## 🐛 استكشاف الأخطاء

### المشكلة: "لا توجد أحداث في قاعدة البيانات"

**السبب:** لم يتم تشغيل `ingest_from_stream.py`

**الحل:**
```bash
python3 ingest_from_stream.py
# تأكد من المسارات:
# LOG_PATH = "/mnt/c/Users/ASUS/rubaa/qemu.log"
# DB_PATH = "/mnt/c/Users/ASUS/rubaa/events.db"
```

### المشكلة: قاعدة البيانات تحتوي على بيانات قديمة

**الحل:**
```bash
# احذف DB القديمة
rm /mnt/c/Users/ASUS/rubaa/events.db

# أعد تشغيل المعالج والواجهة
python3 ingest_from_stream.py
./MainGUI
```

### المشكلة: الواجهة بطيئة جداً

**السبب:** قد تكون هناك ملايين الأحداث

**الحل:**
```bash
# في ingest_from_stream.py، عدل الفلتر:
def is_important_fs_event(event: dict) -> bool:
    """أضف شرط أكثر انتقائية"""
    if event.get("fs_type") in [3, 4, 5]:  # فقط HIT, MISS, FILL
        return True
```
